## 1. Why Async?

Async solves one problem: **waiting efficiently**. When your program waits for a network response or a file read, the CPU sits idle. Async lets it do other work instead.

```
Synchronous:                    Async:
────────────────────            ────────────────────
Request A  ──────►              Request A  ──────►
Wait...                         Request B  ──────►  ← starts immediately
Wait...                         Response B ◄──────
Response A ◄──────              Response A ◄──────
Request B  ──────►              Total: ~1 round trip
Response B ◄──────
Total: ~2 round trips
```

Use async for I/O-bound work — network requests, file operations, database queries. For CPU-bound work, use threads or `rayon` instead.

## 2. Futures and `async`/`.await`

An `async fn` returns a **Future** — a value representing a computation that hasn't run yet. Nothing executes until you `.await` it:

```rust
async fn greet(name: &str) -> String {
    format!("Hello, {name}!")
}

// A Future is lazy — this does nothing yet:
let future = greet("Alice");

// .await drives it to completion:
let result = future.await; // "Hello, Alice!"
```

Rust's async is **runtime-agnostic** — the language provides the syntax but you need a runtime to execute futures. Add **Tokio**, the most widely used one:

```toml
[dependencies]
tokio = { version = "1", features = ["full"] }
```

```rust
#[tokio::main]  // sets up the runtime and runs main as an async fn
async fn main() {
    let msg = greet("Alice").await;
    println!("{msg}");
}
```

Every `.await` is a **yield point** — where the task can pause and let the runtime run other work. This is how concurrency happens without threads.

## 3. Running Tasks Concurrently

**`tokio::join!`** runs multiple futures at the same time and waits for all of them:

```rust
use tokio::time::{sleep, Duration};

async fn fetch(id: u32) -> String {
    sleep(Duration::from_millis(100)).await; // simulates network delay
    format!("result-{id}")
}

#[tokio::main]
async fn main() {
    // Sequential — takes ~300ms total
    let r1 = fetch(1).await;
    let r2 = fetch(2).await;
    let r3 = fetch(3).await;

    // Concurrent with join! — takes ~100ms total
    let (r1, r2, r3) = tokio::join!(fetch(1), fetch(2), fetch(3));

    println!("{r1}, {r2}, {r3}");
}
```

**`tokio::spawn`** runs a task independently in the background:

```rust
#[tokio::main]
async fn main() {
    let handle = tokio::spawn(async {
        sleep(Duration::from_millis(50)).await;
        42
    });

    println!("doing other work while task runs...");
    let result = handle.await.unwrap(); // wait and collect the return value
    println!("task returned: {result}");
}
```

**`tokio::select!`** races futures and takes the first to finish:

```rust
#[tokio::main]
async fn main() {
    tokio::select! {
        result = fetch(1) => println!("fetch finished: {result}"),
        _      = sleep(Duration::from_millis(50)) => println!("timed out"),
    }
}
```

## 4. Channels

Channels let tasks communicate safely. `mpsc` (multi-producer, single-consumer) is the most common:

```rust
use tokio::sync::mpsc;

#[tokio::main]
async fn main() {
    let (tx, mut rx) = mpsc::channel::<String>(32);

    for i in 0..3 {
        let tx = tx.clone();
        tokio::spawn(async move {
            sleep(Duration::from_millis(10 * i)).await;
            tx.send(format!("task {i} done")).await.unwrap();
        });
    }

    drop(tx); // close the last sender so rx knows when to stop

    while let Some(msg) = rx.recv().await {
        println!("{msg}");
    }
}
```

| Channel | Use case |
|---------|---------|
| `mpsc` | Many senders → one receiver |
| `oneshot` | Single send, single receive (request/response) |
| `broadcast` | One sender → many receivers |
| `watch` | Share the latest value (config updates) |

## 5. Putting It Together

A complete async program that fetches multiple URLs concurrently, times out slow ones, and prints a summary — all in one file:

```rust
use std::time::{Duration, Instant};
use tokio::time::timeout;

#[derive(Debug)]
struct CheckResult {
    url:     String,
    status:  Result<u16, String>,
    elapsed: Duration,
}

async fn check(url: String) -> CheckResult {
    let start = Instant::now();

    // Wrap the fetch in a 3-second timeout
    let status = timeout(Duration::from_secs(3), fetch_url(&url))
        .await
        .unwrap_or_else(|_| Err("timed out".into()));

    CheckResult { url, status, elapsed: start.elapsed() }
}

async fn fetch_url(url: &str) -> Result<u16, String> {
    // In production: use reqwest::get(url).await
    // Here we simulate a response
    tokio::time::sleep(Duration::from_millis(80)).await;
    if url.contains("fail") {
        Err("connection refused".into())
    } else {
        Ok(200)
    }
}

#[tokio::main]
async fn main() {
    let urls = vec![
        "https://rust-lang.org".to_string(),
        "https://crates.io".to_string(),
        "https://fail.example.com".to_string(),
    ];

    // Spawn all checks concurrently
    let handles: Vec<_> = urls
        .into_iter()
        .map(|url| tokio::spawn(check(url)))
        .collect();

    // Collect results as they complete
    let mut up = 0;
    let mut down = 0;

    for handle in handles {
        let r = handle.await.unwrap();
        match &r.status {
            Ok(code) => {
                println!("✓ {:3}  {:>5}ms  {}", code, r.elapsed.as_millis(), r.url);
                up += 1;
            }
            Err(e) => {
                println!("✗ ERR  {:>5}ms  {}  ({})", r.elapsed.as_millis(), r.url, e);
                down += 1;
            }
        }
    }

    println!("\nUp: {up}  Down: {down}");
}
```

**Exercise 1 — Concurrent countdown**

Spawn 5 tasks, each sleeping for a different duration (100ms, 200ms, …, 500ms) and printing its ID when done. Use `join_all` from the `futures` crate to wait for all of them, then print the total elapsed time. Confirm it's closer to 500ms than 1500ms.

**Exercise 2 — Async pipeline**

Write an async function `process_all(inputs: Vec<String>) -> Vec<Result<usize, String>>` that takes each string, trims it, rejects empty strings with `Err("empty")`, and returns `Ok(len)` for the character count. Process all inputs concurrently with `tokio::spawn`. Use `?` inside a helper async function for the per-item logic.

## 7. Summary

| Concept | Key Takeaway |
|---------|-------------|
| Future | A lazy computation — nothing runs until `.await` |
| `async fn` | Returns a `Future`; body runs when awaited |
| Tokio | The runtime; `#[tokio::main]` sets it up |
| `join!` | Run multiple futures concurrently, wait for all |
| `spawn` | Launch a task independently; collect via `JoinHandle` |
| `select!` | Race futures — proceed when the first finishes |
| `mpsc` channel | Safe message passing between tasks |
| `timeout()` | Enforce a time limit on any future |
