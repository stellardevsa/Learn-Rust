## 1. `panic!` vs `Result`

`panic!` crashes the program immediately. Use it for bugs — things that should never happen if the code is correct:

```rust
// Good uses of panic
assert!(index < len, "index out of bounds — this is a bug");
let val = map.get("key").expect("key must exist — guaranteed by setup");
```

For anything that can legitimately fail at runtime — bad input, missing files, network errors — return a `Result` instead so the caller can decide what to do:

```rust
// Result is defined as:
enum Result<T, E> {
    Ok(T),   // success
    Err(E),  // failure
}
```

The compiler forces you to handle `Result`. You can't silently ignore it the way you can in most other languages.

## 2. `Result` and `?`

```rust
use std::num::ParseIntError;

fn parse_age(s: &str) -> Result<u32, ParseIntError> {
    let n = s.trim().parse::<u32>()?; // ? returns Err early if parsing fails
    Ok(n)
}

fn main() {
    match parse_age("  25  ") {
        Ok(age) => println!("age: {age}"),
        Err(e)  => println!("invalid: {e}"),
    }

    // Common methods — same pattern as Option:
    let age = parse_age("25").unwrap_or(0);           // fallback on error
    let doubled = parse_age("25").map(|n| n * 2);     // Ok(50) or Err
}
```

`?` is syntactic sugar for "return `Err` early". Compare:

```rust
// Without ?
fn read_file(path: &str) -> Result<String, std::io::Error> {
    let content = match std::fs::read_to_string(path) {
        Ok(s)  => s,
        Err(e) => return Err(e),
    };
    Ok(content.to_uppercase())
}

// With ? — identical behaviour, far cleaner
fn read_file(path: &str) -> Result<String, std::io::Error> {
    let content = std::fs::read_to_string(path)?;
    Ok(content.to_uppercase())
}
```

> **`?` in `main`:** To use `?` directly in `main`, change its signature to `fn main() -> Result<(), Box<dyn std::error::Error>>`.

## 3. Custom Error Types

Define your own error enum to give callers meaningful, matchable codes:

```rust
use std::fmt;

#[derive(Debug)]
enum AppError {
    NotFound(String),
    InvalidInput { field: String, reason: String },
    Io(std::io::Error),
}

impl fmt::Display for AppError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            AppError::NotFound(name)              => write!(f, "{name} not found"),
            AppError::InvalidInput { field, reason } => write!(f, "{field}: {reason}"),
            AppError::Io(e)                       => write!(f, "IO error: {e}"),
        }
    }
}

// Implement From so ? can auto-convert io::Error into AppError
impl From<std::io::Error> for AppError {
    fn from(e: std::io::Error) -> Self { AppError::Io(e) }
}

fn load_config(path: &str) -> Result<String, AppError> {
    let content = std::fs::read_to_string(path)?; // io::Error → AppError::Io via From
    if content.is_empty() {
        return Err(AppError::InvalidInput {
            field: "config".into(),
            reason: "file is empty".into(),
        });
    }
    Ok(content)
}
```

> **Shortcut:** The [`thiserror`](https://docs.rs/thiserror) crate eliminates the `Display` and `From` boilerplate via `#[derive(Error)]`. Worth adding once you're comfortable with the manual version.

## 4. Modules

Rust's module system controls what's public and what's private.

```rust
mod auth {
    // Private by default — only visible inside auth
    fn hash(password: &str) -> String {
        format!("hashed:{password}") // simplified
    }

    // pub — visible outside the module
    pub fn login(user: &str, password: &str) -> Result<String, String> {
        if user.is_empty() { return Err("empty username".into()); }
        Ok(hash(password))
    }

    pub mod roles {
        pub const ADMIN: &str = "admin";
    }
}

fn main() {
    println!("{:?}", auth::login("alice", "secret"));
    println!("{}", auth::roles::ADMIN);
    // auth::hash("x");  // ❌ private
}
```

`use` brings paths into scope so you don't repeat the full path every time:

```rust
use auth::login;
use auth::roles::ADMIN;
use std::collections::HashMap;
use auth::login as sign_in; // alias to avoid name conflicts
```

| Modifier | Visible to |
|----------|-----------|
| (nothing) | Current module only |
| `pub` | Everywhere |
| `pub(crate)` | Anywhere in this crate |
| `pub(super)` | Parent module only |

**Multi-file modules** follow the directory structure. `mod foo;` in `main.rs` tells Rust to look for `src/foo.rs` (or `src/foo/mod.rs`). Everything else works the same — `pub` controls what's exported, `use` imports it.

Split this code into a `math` module with a private `validate` helper and a public `divide` function. Add a `math::constants` sub-module with a public `PI` constant. Confirm that calling the private helper directly from `main` is a compile error.

```rust
fn validate(n: f64) -> bool { n != 0.0 }
fn divide(a: f64, b: f64) -> Result<f64, String> {
    if !validate(b) { return Err("division by zero".into()); }
    Ok(a / b)
}
```

## 6. Summary

| Concept | Key Takeaway |
|---------|-------------|
| `panic!` | For bugs — not for expected failures |
| `Result<T, E>` | The primary error channel; compiler forces you to handle it |
| `?` | Propagates `Err` early — keeps code linear |
| Custom errors | Enum + `Display`; implement `From` for automatic `?` conversion |
| `mod` / `pub` | Declare modules and control visibility |
| `use` | Bring paths into scope; alias with `as` |
| Multi-file | `mod foo;` links `src/foo.rs`; same visibility rules apply |
